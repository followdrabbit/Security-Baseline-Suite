Project: Aureum
Product: Security Baseline Suite
Date: 2026-04-12

System context for LLM assistants:

1. Backend/storage model
- The app runs with a local SQLite backend.
- Main API entrypoint: local-api/server.mjs
- Database file: local-api/data/security-baseline.sqlite
- Client integration module: src/integrations/localdb/client.ts
- Type definitions: src/integrations/localdb/types.ts

2. Runtime mode
- Start everything with npm run dev:local
- Local API URL expected by frontend: http://127.0.0.1:8787
- Auth is local session based (username/password).
- Public sign-up and OAuth sign-in are disabled in local mode.
- Default bootstrap user: admin / Admin@123456 (password change required on first login).
- Admin can create local users in Settings -> Usuarios locais.
- Every user created by admin must change password on first login.

3. AI provider configuration
- Supported providers in AI Integrations: OpenAI, Google Gemini, Anthropic Claude, Grok/xAI, and Ollama local models.
- Provider settings are persisted per user in `ai_provider_configs`.
- Provider registry is persisted in `ai_provider_catalog` and model registry in `ai_provider_models` (per user scope).
- AI Integrations supports full CRUD for providers and models (create/read/update/delete).
- Primary model and fallback model are configurable per provider.
- Advanced model parameters are optional, disabled by default, and enabled per parameter flag.
- API keys are encrypted at rest by local-api before storage in SQLite.
- Connection testing should use /functions/v1/test-ai-provider (server-side), not direct browser calls.

4. Removed components
- Supabase code paths were removed from app runtime.
- Do not introduce new Supabase imports or env vars.

5. Testing
- Unit/integration tests: npm test
- Coverage: npm run test:coverage
- Tests mock localDb (not supabase).
- Latest snapshot (2026-04-12): 165 tests passing, 47.67% statements, 60.47% branches.

6. Documentation alignment
- README.md and docs/* reflect local SQLite mode.
- In-app docs (Documentation page + translations) must stay aligned with the local auth flow.
- Keep docs consistent with local-api and localdb integration.
- AI documentation must cover provider/model CRUD, fallback model behavior, and advanced-parameter flags.


7. UX/i18n and AI gating behavior
- Core pages/components use i18n keys for EN/PT/ES labels, statuses, toasts, and navigation text.
- Flows that require AI (source extraction, pipeline execution, project-to-AI operations) must validate provider setup first.
- If no provider is configured, surface the explicit warning from `AI_CONFIGURATION_REQUIRED_MESSAGE` / `common.aiIntegrationRequired`.

8. Local auth password policy
- Local passwords must follow server policy: minimum 12 chars, max 128, at least 1 uppercase, 1 lowercase, 1 number, 1 special char, and no spaces.
- Default bootstrap credentials remain `admin / Admin@123456` for first login in distributed code.
